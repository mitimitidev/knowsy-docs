2.1 — Recommend frontend & backend architecture SoW §2

Recommended direction — MVP

  • Runtime: Bun — one fast toolchain for install, run, bundle & test; native TypeScript.
  • Framework: Next.js 16 (App Router) — React Server Components by default, streaming, and a mature ecosystem.
  • Rendering: static for marketing pages, server-rendered for address search, streamed for report generation, client interactivity only where needed.
  • API layer: co-located in the same Next.js app (Route Handlers + Server Actions) — one service, one deploy for the MVP; carved out later if a flow needs to scale independently.
  • Validation: Zod — schemas shared across forms, API boundaries, and external/OCR data ingestion (single source of truth for types).
  • Data layer: Drizzle ORM over Postgres — typed SQL, lightweight checked-in migrations.
  • Auth & payments: hosted providers integrated in-app (provider selection & flow assumptions captured in 1.3 log).

2.2 — Design data storage & report generation flow SoW §2

Data model — Postgres (Neon) via Drizzle

  • Identity: user, session, account — owned by Better Auth, in our schema.
  • Domain: property (normalized address + parcel id), report (status, owner, property ref, version), order (Stripe ref, amount, status).
  • Pipeline: document (source deed → Vercel Blob key, type, checksum), extraction (OCR output + confidence, linked to document), source_fetch (vendor response cached as JSONB, with provenance, fetched_at + TTL), review_item (low-confidence flags awaiting human sign-off).
  • Storage split: structured data in Postgres; raw vendor payloads in JSONB; binary deed files in Vercel Blob (signed URLs, keys only in DB).
  • Cache & reuse: fetches/extractions keyed by property are reused across purchases while within TTL; stale → refetch. Reports pin the source_fetch versions they were built from (point-in-time provenance).
  • PII: deeds carry owner names — encryption at rest, row-level access scoped to owning user, retention policy on source files.

Report generation flow — end to end

  • Address search → resolve & normalize → property record.
  • Checkout (Stripe) → order paid → create report (status: pending).
  • Enqueue durable workflow keyed by reportId (idempotent).
  • Steps: fetch data sources (parallel; reuse cached within TTL) → OCR deeds (upload → extract) → normalize + confidence-score → assemble → persist. Low-confidence items → review_item, report ships partial.
  • UI polls / streams status: loading → partial → success (or error), each step's state in Postgres.
  • On completion: report available; notify user.

Queue / job strategy — Inngest

  • Durable multi-step functions with per-step retries & observability.
  • Idempotency keyed by reportId; per-vendor concurrency limits to respect rate caps.
  • Low-confidence extractions routed to a manual-review state rather than failing the report.
  • Step state mirrored to Postgres so the UI never depends on the queue being reachable.

2.3 — Plan infrastructure, scalability & observability SoW §2

Stateless service — design principles

  • No state in the request path: the Next.js web/API tier holds nothing durable in memory — no in-process queues, sticky sessions, or local caches. Any instance can serve any request, so Vercel can scale instances horizontally and recycle them freely.
  • All durable state is externalized: domain & step state → Postgres (Neon); binary deeds → Vercel Blob; orchestration & queue state → the workflow engine. The compute tier is disposable.
  • Sessions are token-backed: auth via signed cookies over Postgres-stored sessions — no server affinity, no sticky load balancing.
  • Enqueue-and-return: the API validates, creates the order/report, kicks off a durable run keyed by reportId, and returns immediately. All long work happens in durable steps, never inside an HTTP request.
  • Idempotent by construction: runs keyed by reportId; every step is safe to retry/replay (upserts, checksums, dedup on source_fetch) so autoscaling, retries, and cold starts never double-charge or double-write.
  • Connections don't fan out: Neon serverless driver (HTTP) so scaling to many instances never exhausts Postgres connections.
  • Backpressure lives in the engine, not the user: per-vendor concurrency/rate limits are enforced by the orchestrator (below), keeping us under external caps regardless of inbound traffic spikes.

Recommendation

Pick · MVP

Inngest. The pipeline's real ceiling is external vendor rate limits + OCR throughput (established in 2.1–2.2), and Inngest solves exactly that with first-class per-key concurrency, throttling, and rate-limiting — zero custom code — while the compute stays fully stateless on Vercel. Lowest-risk, lowest-ops path to a scalable service.

Fast-follow

Vercel Workflows. Strongest "one platform, one deploy" story and removes a second dashboard/billing relationship. Revisit once we want to collapse vendors — main gap today is turnkey per-vendor flow control, which we'd otherwise hand-roll. Portable design (below) keeps this swap cheap.

Defer

Temporal. Best-in-class for very complex, long-running orchestration at scale — but it requires always-on worker processes and a cluster/Temporal Cloud, which breaks the stateless-serverless model and adds the most ops. Reserve as an "extract later" target if one flow's complexity outgrows serverless.

Keeping the orchestrator swappable

  • Thin port, not a hard dependency: the app calls an internal jobs.enqueueReport(reportId) / step interface; the Inngest binding lives behind it. Swapping to Vercel Workflows or Temporal is an adapter change, not a rewrite.
  • Engine owns orchestration, Postgres owns truth: step state is mirrored to Postgres (2.2) so the UI and recovery never depend on which engine is wired in.
  • Idempotent steps + reportId keying are engine-agnostic — they work the same under all three.

Infrastructure & scalability

  • Compute: Vercel autoscaling functions for web + API; the orchestrator (Inngest) absorbs pipeline load and enforces per-vendor concurrency limits.
  • Database: Neon autoscaling + scale-to-zero; serverless driver (HTTP/WS) to avoid connection exhaustion under serverless fan-out.
  • Real ceiling = external vendors: their rate limits + OCR cost/throughput, not our infra. Cache-and-reuse (2.2) + engine-side throttling are the levers that keep us under those caps.
  • Files: Vercel Blob behind CDN; serve derived/thumbnail assets, not raw scans, to control egress.
  • Environments: preview-per-PR (Vercel preview + Neon branch) → Production; promote on merge.

Observability & monitoring

  • Errors: Sentry across frontend, API, and workflow steps.
  • Pipeline: the orchestrator dashboard (Inngest) as first-class step tracing / retries; step state also mirrored to Postgres so the UI never depends on the engine being reachable.
  • Logs & metrics: Vercel + log drain; Neon DB metrics & slow queries; uptime/synthetics on the critical path.
  • Business KPIs: reports generated, success/partial/error rates, cost per report, review-queue depth, vendor latency & error rates.

Key trade-offs (decision framework)

  • Stateless serverless over persistent workers: autoscaling & zero idle cost win for an MVP — which is precisely why Temporal's always-on worker model is deferred.
  • Modular monolith now, extract later: simplicity over premature decoupling; split trigger = a flow needing independent scaling/runtime (also the Temporal trigger).
  • Serverless limits: cold starts, connection & execution-time caps — mitigated by the durable engine (long work) + Neon serverless driver.
  • Managed vendors (Vercel/Neon/Inngest): speed now vs lock-in later — hedged by portable choices (standard Postgres + Drizzle, S3-compatible storage, the swappable job port above).
  • Cache TTL: cost & latency vs data freshness/legal exposure — tuned per source in Phase 3.
