Frontend and backend architecture recommendation, data storage strategy, report generation flow, and infrastructure planning for the Knowsy MVP.

Architecture overview

Recommended stack

A monorepo Next.js application deployed on Vercel. The frontend and API routes live together. Background processing (deed retrieval, OCR, AI analysis) runs as Vercel Functions triggered by queue events. Storage is split: Vercel Blob for raw deed PDFs, Postgres (via Vercel Marketplace) for structured report data.

Frontend

Next.js App Router · React · Tailwind CSS · Stripe Elements

Backend

Next.js API routes · Vercel Functions · Fluid Compute

Infrastructure

Vercel (hosting) · Postgres · Blob · Queues (beta)

Report generation flow

End-to-end pipeline

  • User submits address → API validates against assessment roll → creates a report job record
  • Stripe checkout completes → webhook triggers the ingestion queue
  • Queue worker retrieves deed from WRO (or fallback vendor) → stores raw PDF in Blob
  • OCR function reads PDF text → AI function extracts structured fields
  • Chain-of-title loop: each parsed deed provides a prior-deed reference → loop continues until no prior deed found
  • Report assembled from all parsed deeds → stored in Postgres → user notified by email

Key technical decisions

Rendering

App Router over Pages Router

Server components reduce client bundle size. Streaming allows partial report delivery. Consistent with current Next.js direction.

Background jobs

Vercel Queues (beta) over Inngest / external queue

Keeps infrastructure on a single platform. Acceptable risk given the beta label — fallback is Inngest, which has similar DX. Deed retrieval must run async; a synchronous API route would time out.

Database

Postgres over NoSQL

Report data is relational (deeds, owners, encumbrances link to each other). Schema enforcement matters for a legal-adjacent product. Postgres via Vercel Marketplace removes operational overhead.

OCR

External OCR API over self-hosted Tesseract

Deed scans vary widely in quality. Cloud OCR (Google Document AI or AWS Textract) handles handwritten text, stamps, and low-contrast scans significantly better. Cost analysis in Phase 4.

AI model

Claude (Anthropic) for deed parsing

Strong structured-extraction performance on legal-style text. Tool-use / JSON mode produces reliable output schema. Cost modeled against per-report budget in Phase 4.

Observability & reliability

Monitoring

  • Vercel built-in function logs and error tracking
  • Queue dead-letter alerts for failed deed retrievals
  • Report job status exposed to user via polling endpoint

Failure handling

  • WRO retrieval: 3 retries with exponential backoff, then fallback vendor
  • OCR failure: flag report as manual review required
  • Payment: Stripe idempotency keys prevent double-charges on retry
