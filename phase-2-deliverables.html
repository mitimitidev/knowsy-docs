2.1 — Frontend & backend architecture

Runtime

Bun — one fast toolchain for install, run, bundle & test; native TypeScript.

Framework

Next.js 16 (App Router) — React Server Components by default, streaming, mature ecosystem.

API layer

Co-located Route Handlers + Server Actions — one service, one deploy for MVP; carved out later if needed.

Rendering strategy

  • Static rendering for marketing pages
  • Server-rendered pages for address search
  • Streamed content for report generation states
  • Client interactivity only where necessary

Validation

Zod — schemas shared across forms, API boundaries, and external/OCR data ingestion. Single source of truth for types.

Data & ORM

Drizzle ORM over Postgres — typed SQL, lightweight checked-in migrations.

2.2 — Data storage & report generation

Database schema — Postgres / Neon

Identity layer

User, session, and account records managed by Better Auth.

Domain entities

Property records (normalized addresses and parcel IDs), reports (status and ownership), orders (Stripe references).

Pipeline tables

Documents (source deeds stored as Vercel Blob keys), extractions (OCR output with confidence scores), source fetches (vendor responses cached as JSONB with timestamps), review items (flagged low-confidence extractions awaiting human approval).

Storage architecture

Structured data in Postgres; raw vendor payloads in JSONB; binary deed files in Vercel Blob (signed URLs, keys only in DB). Fetches keyed by property and reused across purchases within TTL windows.

PII handling

Deeds carry owner names — encryption at rest, row-level access scoped to owning user, retention policy on source files.

Report generation flow

1

Address normalization

Input validated and matched against assessment roll to resolve owner of record and parcel ID.

2

Payment processing

Stripe checkout — idempotency keys prevent double-charges on retry.

3

Job enqueuing

Payment webhook triggers the Inngest workflow. API returns immediately.

4

Parallel data fetch & OCR

Deed retrieved from WRO (or fallback vendor), stored in Blob, run through OCR and AI parsing in parallel steps.

5

Status polling

Frontend polls a status endpoint every 5 seconds, displaying live step progress to the user.

6

User notification

Report marked complete in Postgres, email sent with report link.

Queue strategy — Inngest

Durable multi-step functions with per-step retries and observability. Idempotency keyed by report ID. Built-in vendor concurrency controls prevent rate-limit breaches.

2.3 — Infrastructure, scalability & observability

Stateless service principles

No state in the request path — the web tier holds no durable memory, allowing Vercel to scale horizontally. All durable state is externalized: domain & step state → Postgres (Neon); binary deeds → Vercel Blob; orchestration & queue state → Inngest.

Authentication uses token-backed sessions via signed cookies over Postgres — no server affinity required.

Scalability patterns

  • Enqueue-and-return: APIs validate and immediately return while durable steps handle long-running work
  • Neon serverless driver (HTTP) prevents connection exhaustion at scale
  • Real ceiling is external vendors: their rate limits + OCR cost/throughput, not our infra

Observability stack

  • Sentry — errors across frontend and backend
  • Inngest — orchestrator visibility and step-level tracing
  • Vercel — infrastructure metrics
  • KPIs: reports generated, success/error rates, cost per report, review-queue depth, vendor latency

Orchestrator evaluation

Option Fit Notes
Inngest First-class per-key concurrency, throttling, rate-limiting — zero custom code Durable, serverless-native, strong observability Pick · MVP
Vercel Workflows Platform consolidation upside Recommended for future fast-follow once mature Fast-follow
Temporal Strong guarantees but always-on workers conflict with serverless model Deferred — adds infra overhead not justified at MVP scale Deferred
Design principle: The orchestrator is wired behind a thin internal interface — jobs.enqueueReport(reportId) — so the adapter can be swapped without rewrites if we migrate from Inngest later.
