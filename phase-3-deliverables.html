3.2 — Document extraction workflow

Deed image → structured data — 8-step process

1

Acquisition

Deed image downloaded from WRO via browser automation and stored in Vercel Blob. Original preserved for audit trail.

2

Preprocessing

Image normalization — deskew, contrast enhancement, resolution upscaling for low-quality historical scans.

3

OCR

Text recognition run on preprocessed image. Word-level confidence scores captured for downstream quality gating.

4

Schema-constrained LLM extraction

Claude Opus 4.8 vision handles complex legal document parsing directly — no county-specific model training required. Extracts grantor, grantee, date, consideration, legal description, encumbrances, and prior-deed reference. Output validated against a shared Zod schema.

5

Normalization

Names, dates, and addresses standardized to canonical formats for consistent chain-of-title linking.

6

Confidence scoring

Composite score from OCR confidence + LLM extraction certainty. Fields below threshold flagged for human review.

7

Persistence

Structured extraction written to Postgres pipeline tables. Raw deed PDF key retained in Blob.

8

Human review queue

Low-confidence extractions surfaced to a review queue. Reviewer can correct fields; corrections feed back to improve future extractions.

Model choice: Claude Opus 4.8 vision handles complex legal documents directly — no county-specific model training, no fine-tuning pipeline to maintain.

3.3 — Pipeline architecture

Inngest-based durable workflow

All data ingestion runs as durable Inngest functions. Sources are fetched in parallel where possible, with per-vendor concurrency limits to avoid rate-limit breaches. Each step has independent retry logic — a vendor failure doesn't block other data sources from completing.

Connector pattern

  • Uniform adapter interface for all 25 sources
  • Each connector: fetch → validate (Zod) → cache → return
  • New sources added without touching pipeline core
  • Graceful degradation when vendor unavailable — report marks that section as incomplete rather than failing entirely

Caching strategy

  • Results cached by property ID in Postgres (JSONB)
  • TTL-based freshness per source type — deed data longer, market data shorter
  • Cache hit reused across multiple purchases for same property
  • Raw vendor payloads stored separately from normalized output

Governance & monitoring

Validation

All pipeline outputs validated against shared Zod schemas — same schemas used at API boundaries and forms. Schema mismatches surface as structured errors, not silent failures.

Metrics tracked

  • Cost per source per report
  • Vendor latency and error rates
  • Extraction accuracy (confidence score distribution)
  • Review queue depth
