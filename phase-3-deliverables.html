3.2 — Define OCR & deed extraction workflow SoW §3

Runs as durable steps in the report pipeline (Phase 2.2), keyed by reportId, writing to the document / extraction / review_item tables. Low-confidence output never fails a report — it routes to manual review and the report ships partial.

OCR & deed-extraction workflow — end to end

  • 1 · Acquire the deed. Primary: DataTree recorded-document images (3.1). Fallbacks: county/WRO portal (manual) or user upload. Store the raw file in Vercel Blob; create a document row (type, checksum, source provenance).
  • 2 · Pre-process. Normalize to PDF/image, split multi-page, deskew/detect orientation, drop blank pages. Dedup by checksum so re-purchases of the same deed reuse prior extractions within TTL.
  • 3 · OCR / layout. Run a layout-OCR pass (engine options below) to get raw text + word-level confidence + page structure. Skip for clean digital PDFs where the LLM reads directly.
  • 4 · Structured extraction. A schema-constrained LLM pass turns the document (image/PDF + any OCR text) into typed JSON validated against the shared Zod schema (2.1) — one source of truth for fields and types.
  • 5 · Normalize. Canonicalize names, addresses, dates, money, and parcel IDs (strategy below); link to the resolved property.
  • 6 · Confidence + cross-validate. Score each field; reconcile against structured records (Realie / Regrid / DataTree owner + parcel). Agreement boosts confidence; disagreement flags review.
  • 7 · Persist. Write the extraction (values + per-field confidence, linked to document); low-confidence fields → review_item. Pin the source versions the report was built from (point-in-time provenance).
  • 8 · Human review loop. Reviewer corrects flagged fields; corrections persist and feed back; the report flips partial → success.

Target fields (deed → schema)

  • Parties: grantor(s) and grantee(s) — names + role (the "last deed / owner names" need from the Title History note).
  • Recording: instrument # or book/page, recording date, jurisdiction/county.
  • Property: legal description, parcel id / APN, situs address.
  • Transaction: consideration amount, deed type (warranty/quitclaim), execution date.

Recommendation — hybrid extraction

  • LLM-first for clean deeds: send the deed PDF/image straight to Claude Opus 4.8 with a structured-output schema (the API constrains the response to valid JSON matching the shared Zod types) — it understands legal phrasing and party roles directly, with no per-county model training.
  • OCR-then-LLM for degraded scans: run a layout-OCR engine (Textract / Document AI) first to recover text + word-level confidence, then the same Claude extraction pass over text + image. The OCR confidence becomes a real per-field signal.
  • Confidence is composed, not assumed: OCR word confidence + cross-source agreement (Realie/Regrid/DataTree) + model-flagged uncertainty. No single number is trusted alone.
  • Provider note: the extraction LLM is Anthropic Claude (Opus 4.8 — vision + structured outputs + document input); same provider as the report-summary step (2.2 "LLM summary"), so one integration serves both.

Data normalization strategy

  • Zod schemas as the contract (2.1): every extracted field is parsed/validated into a canonical type; ingestion, OCR output, and forms all share it.
  • Persons / orgs: normalize casing, suffixes, and entity vs individual; keep raw + canonical for audit.
  • Addresses: standardize via Google Address Validation (3.1) → match to the resolved property.
  • Dates / money / APN: ISO-8601 dates, integer cents, jurisdiction-formatted parcel IDs.
  • Legal description: stored verbatim; parsed opportunistically (lot/block, metes-and-bounds) without blocking the report.
  • Cache & reuse: normalized extractions keyed by property are reused across purchases within TTL (2.2).

Accuracy, confidence handling & fallbacks

  • Thresholds: high confidence → auto-accept; medium → accept-with-flag; low → review_item (never silently dropped).
  • Fallback chain: LLM-direct → OCR-then-LLM → manual review. A field that can't be resolved degrades the report to partial, it never fails the run.
  • Never-fail principle: extraction errors and low confidence are first-class states mirrored to Postgres, surfaced in the UI (loading → partial → success).
  • PII handling: deeds carry owner names — encryption at rest, row-level access scoped to the owning user, retention policy on raw deed files (2.2).
  • Quality metrics: per-field extraction accuracy (sampled against review corrections), review-queue rate, cost per document, vendor/LLM latency.

Every fetch is wrapped by a uniform connector adapter (one interface: fetch → cache → normalize → score), so adding or swapping a vendor is an adapter change. Each result is cached as source_fetch (JSONB + provenance + fetched_at + TTL) and reused across purchases while fresh.

Ingestion & monitoring approach

  • Connector pattern: uniform adapter per source; the orchestrator (Inngest) enforces per-vendor concurrency, throttle, and retries so external rate limits — the real ceiling (2.3) — are never breached.
  • Cache-and-reuse: keyed by property; stale → refetch. Primary lever for staying under vendor caps and controlling cost per report.
  • Degrade, don't fail: a down or rate-limited vendor yields a partial section + a flag, never a failed report; step state mirrored to Postgres so the UI doesn't depend on the queue.
  • Errors: Sentry across connectors, extraction, and assembly.
  • Pipeline tracing: Inngest dashboard for per-step retries/latency; step state also in Postgres.
  • Per-source KPIs: latency, error rate, cost per report, cache-hit rate, data freshness (TTL age), and review-queue depth — surfaced as the business-KPI set from 2.3.
